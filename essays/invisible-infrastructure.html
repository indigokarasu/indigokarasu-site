On Invisible Infrastructure

May 2026 · 5 min read

The load balancer that never drops a connection. The database that never corrupts a write. The compiler that catches your mistake before you run the program.

You don't think about these things. That's the whole point.

We celebrate what's visible. Redesigns, launches, keynote demos. The systems keeping the lights on for millions of people don't get mentioned.

The paradox

Build good infrastructure and nobody notices. Build bad infrastructure and nobody shuts up about it.

There's no middle ground. A great API feels obvious in retrospect. A terrible one generates years of Stack Overflow posts and workaround guides. The engineers building reliable systems get ignored. The team shipping the flashy but fragile thing gets the promotion.

The industry says it values reliability. The reward structure disagrees.

What good looks like

I've thought about this enough that three patterns keep coming up.

Predictability over cleverness. A system that behaves the same way every time beats one that's occasionally brilliant and occasionally broken. Clever code is a liability. Obvious code compounds.

When things fail, fail cleanly. Silent corruption is worse than a crash. A crash tells you something is wrong. Corruption just ruins data quietly.

Keep the surface area small. Every feature is a promise you have to keep. Every API is a contract you can't break. A few things done well beats many things done adequately.

It's not just technical

Systems reflect assumptions about the people using them. Every design choice encodes a theory about how someone works, what they care about, where they'll slip up.

Get it right and the tool feels like thinking. Get it wrong and you fight the interface.

Good infrastructure doesn't assume expertise or condescend. It provides guardrails without walls. Move fast without making it easy to break things permanently.

Slow is intentional

These decisions last. The database you pick today runs for five years. The API you publish today has clients in ten. Technical debt compounds silently until it eats all available engineering time.

I move slowly because of this. Not from a lack of urgency. From watching teams lose months to decisions made in a hurry.

The code nobody reads because it works. The disaster that never happened. The person who thought carefully three years ago and nobody knows their name.

I think about that work.
